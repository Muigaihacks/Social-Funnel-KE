Part 0: Rationale — Why We Build This Ourselves First
Before a single module is configured in Make.com, this needs to be said plainly:
  



Why Make.com (Not Zapier, Not n8n)
Operations-based pricing means you pay per action, not per task. A 12- step automation costs the same 12 operations regardless of complexity. For a system with branching logic and multi-step sequences, this is significantly cheaper than Zapier.  
Visual scenario builder maps directly to the flowcharts in this document. Every person on the team can read and modify a Make.com scenario without writing code.
Native error handling, retry logic, and partial execution prevent data loss when external APIs fail — critical for a client-facing system.
Task-based pricing means a 12-step Zap costs 12 tasks. At scale with multiple scenarios running, costs compound fast. More importantly,  
Zapier's logic capabilities (filters, routers, iterators) are weaker for complex branching flows.
n8n self-hosted is powerful but introduces infrastructure overhead (servers, uptime, SSL, maintenance). At this stage, Make.com's managed infrastructure removes operational risk. Revisit n8n when monthly Make.com costs exceed R8,000/month.  




Why Airtable as CRM (Not HubSpot Free, Not Google Sheets)
HubSpot Free is tempting but the API limits on the free tier make automation-heavy use painful. Paid HubSpot is expensive before you have clients. Google Sheets breaks at scale, has race conditions with concurrent writes, and has no relational data model.
Airtable gives you a relational database with a clean API, native Make.com integration, visual views (Kanban pipeline, calendar, gallery), and a free tier that handles up to 1,200 records — sufficient for the first 6 months. When you outgrow it, migrating to HubSpot or a proper CRM is straightforward because your data model is already clean.


Why GPT-4o-mini for AI (Not GPT-4o)
  



The Self-Eat-Your-Own-Cooking Principle
Every scenario in this document has two versions:
* The Social Funnel internal version: running on our own leads, our own pipeline
* The client template version: the same scenario, parameterised for a construction firm client


We build and run the internal version first. We fix the bugs on our own pipeline before our clients experience them. When we hand a client the template, it is battle-tested.

   2. Airtable (CRM Foundation)
5. Create account at airtable.com with your business email.
6. Create a new Base called 'Acquisition OS — Social Funnel'.
7. Create the following tables (covered in full in Part 3):
   * Contacts (all leads and clients)
   * Pipeline Stages
   * Message Log (every message sent)
   * Audit Bookings
   * Follow-Up Queue
   * Reactivation Queue
8. Generate your Airtable API key: Account > Developer Hub > Personal Access Tokens > Create Token. Permissions needed: data.records:read, data.records:write, schema.bases:read.




   3. OpenAI

10. Add a payment method. Set a monthly spend limit of R300 to prevent runaway costs.
11. Generate an API key: API Keys > Create new secret key. Name it 'Make-AcquisitionOS'.
12. Note your Organisation ID from Settings — needed for some API calls.  


   4. WhatsApp Business API — 360dialog
   13. Go to 360dialog.com and create a business account.
   14. You will need a dedicated phone number for Social Funnel (not your personal number). A local South African number works. Get a SIM for this purpose.

   16. Once approved, generate your 360dialog API key from the dashboard.
   17. Note your WABA ID (WhatsApp Business Account ID) — needed in every API call.

S1: Lead Capture Router
  



Module-by-Module Build
      39. MODULE 1 — Custom Webhook: Add > Webhooks > Custom Webhook. Copy the URL. This is your universal lead intake endpoint. Give it to every source (Facebook, Tally form, website form). Name it: 'AcqOS-Lead-Intake'.
      40. MODULE 2 — Set Variable: Extract and normalise fields from the webhook payload. Map: Name
= {{1.name || 1.full_name || 1.Name}}, Phone = {{1.phone || 1.telephone}}, Email = {{1.email}}, Source = {{1.source}}, Company = {{1.company || 1.firm_name}}.
      41. MODULE 3 — Airtable: Search Records. Table: Contacts. Filter: Phone = {{2.Phone}} OR Email =
{{2.Email}}. This checks for duplicates. Operations: 1.
      42. MODULE 4 — Router (Condition): Branch A = Record exists (update). Branch B = Record does not exist (create).
      43. MODULE 4A — Airtable: Update Record (if duplicate). Fields to update: Last Contact Date = now(), Lead Source = append new source if different.
      44. MODULE 4B — Airtable: Create Record (if new). Map all fields. Set Pipeline Stage = 'New Lead'. Set First Contact Date = now(). Set WhatsApp Opt-In = true (form submission implies consent — include opt-in language on all forms).
      45. MODULE 5 — Set Variable: Store the Airtable Record ID from whichever branch ran. This is passed to S2 and S3.
      46. MODULE 6 — HTTP Module: POST to S2's webhook URL with Record ID, Name, Phone, Email, Company, Source. This triggers S2 immediately.

      52. MODULE 6 — Airtable: Create Message Log entry. Contact = {{1.RecordID}}, Channel = WhatsApp, Type = First Response, Body = {{2.choices[0].message.content}}, Sent At = now().

Part 5: Credit Optimisation — Building a Lean System
Make.com charges per operation (each module execution = 1 operation). A poorly built scenario can use 20+ ops for something achievable in 5. These principles keep your system lean.


The 10 Rules of Make.com Credit Efficiency
         106. BATCH AIRTABLE WRITES: Instead of updating 3 fields with 3 separate Airtable modules, combine them into one Update Record module. 3 ops becomes 1 op.
         107. USE FILTERS, NOT ROUTERS: A Router creates branches and charges operations for each branch evaluation. A Filter (built into the connection line) stops the flow for free if the condition is false. Use Filters for simple yes/no conditions.
         108. CACHE AI RESPONSES: If you call OpenAI to generate a welcome message for a lead from 'Facebook Ad', save that response in Airtable. The next Facebook Ad lead gets the same template (with name substituted), not a fresh AI call.
         109. SEPARATE SCENARIOS BY FREQUENCY: A scenario that runs every minute (watching for leads) costs more than one that runs every 15 minutes. Set polling scenarios to the minimum frequency you actually need. Most follow-up timing is day-level — 15-minute polling is fine.
         110. USE INSTANT TRIGGERS: Webhooks (instant) cost the same as scheduled polling but fire only when needed. Replace all polling scenarios with webhooks where possible. S1 through S9 all use webhooks — only S10 (weekly report) uses scheduling.
         111. MINIMISE AIRTABLE READS: Before reading a record, ask: do you already have this data from an earlier module in the same scenario? Store it in a Set Variable module and reference it downstream. Avoid re-fetching records you already have.
         112. SLEEP MODULES ARE FREE: Make.com's Sleep module costs 0 operations. Use it for timed delays within a scenario (e.g., sending a second message 30 seconds after the first) instead of creating a new scheduled scenario.
         113. BUNDLE NOTIFICATIONS: Instead of sending a Slack message from S2, S3, and S4 separately, accumulate the data and send one rich Slack message from S9. Fewer scenario triggers = fewer base operation charges.
         114. USE MAKE.COM'S BUILT-IN FUNCTIONS: Date math, text parsing, and conditional logic done in Make.com's formula syntax costs 0 operations. The same done via an HTTP call or a custom function costs operations. Learn the formula library.
         115. MONITOR YOUR OPERATION DASHBOARD WEEKLY: Make.com shows operations used per scenario per day. A scenario that suddenly uses 10x its normal ops has a bug (likely an infinite loop or missing stop condition). Check this every Sunday alongside your pipeline review.

         * Open with: 'Before we start, I want to be upfront — this 15 minutes is not a pitch. It is a diagnosis. I am going to ask you 3 questions, and we will know inside 5 minutes whether there is a real problem here worth fixing.'

Part 7: Testing Protocol — Before Going Live
Do not go live on real leads until you have completed this testing protocol. Every scenario should be tested in sequence using a test contact (your own phone number and email).


Testing Sequence
         116. STEP 1 — Test S1: Submit a test form to your S1 webhook URL using a tool like Postman or hurl.it. Verify: Airtable record created, all fields populated correctly, HTTP call to S2 and S3 fired.
         117. STEP 2 — Test S2: Confirm you receive a WhatsApp message on your test number within 60 seconds. Confirm email received. Check Airtable: Pipeline Stage = Contacted, Last Contact Date updated.
         118. STEP 3 — Test S3: Confirm Lead Score populated in Airtable. Check Slack #hot-leads for alert (if score >= 7). Confirm S4 or S7 triggered appropriately.
         119. STEP 4 — Test S4: Manually advance Follow-Up Count in Airtable and trigger S4 for each touch. Verify: correct message sent via correct channel, Airtable updated, queue entry marked Sent.
         120. STEP 5 — Test S5: Make a real Calendly booking using your test lead's email. Verify: confirmation WhatsApp received, Airtable updated, S4 queue cleared (Status = Skipped), Slack #bookings alert received.
         121. STEP 6 — Test S6: Manually trigger no-show webhook or use Calendly's test event. Verify: recovery message within 5 minutes, Slack alert, Airtable updated.

         123. STEP 8 — Test S10: Run S10 manually. Verify: Slack report received, Google Sheets row appended.
         124. STEP 9 — End-to-End Test: Start from a fresh test submission (S1) and let the system run for 24 hours. Observe every automated action. Check every Airtable field. Fix anything unexpected.
         125. STEP 10 — Soft Launch: Run on REAL leads for 7 days with daily manual review of every Airtable record. Only after 7 clean days does the system run without daily monitoring.

The Client Version Architecture
Each client gets their own Airtable Base (not a shared table — separation protects client data). Each client gets their own Make.com Team with their scenario copies. The scenarios are identical to our internal ones, parameterised with their API keys and business details.
What Changes Per Client
         * Airtable Base ID and API key (their data, not ours)

         * Calendly API key (their booking link)
         * OpenAI prompt context (their company name, vertical, USP)
         * Slack workspace (their internal team notifications)
         * Lead source webhooks (their website form, their Facebook Page)




What Does NOT Change Per Client
         * The scenario logic — identical flow, just different API connections

         * The 7-touch sequence timing and structure
         * The reporting structure and Airtable data model
